## What Is This?

Diablo II is a 32-bit Windows ARPG released by Blizzard Entertainment in 2000.
Its source code was never publicly released. Over the past two decades, the
modding and emulation community has reverse engineered the game's internals
through disassembly, runtime memory analysis, and save file archaeology.

**D2RE** packages the accumulated knowledge of that community into a single,
coherent, well-documented toolkit. It covers:

- **The complete binary protocol** — every packet sent between the D2 game
  client and server, with field-level definitions and plain-English explanations
  of what triggers each packet and what it means in game terms.

- **The save file format** — Diablo II uses a custom bit-packed binary format
  for `.d2s` character files. This toolkit includes a full parser that reads
  every field: stats, skills, inventory items with all affix data, mercenary
  equipment, quest flags, and waypoints.

- **The item generation pipeline** — Diablo II's item drop system is
  deterministic and reproducible. Given a seed and an item level, the same
  affix rolls will occur every time. This toolkit implements the full 6-stage
  pipeline: Treasure Class resolution, item level math, quality determination
  with Magic Find, affix selection, unique/set matching, and socket rolling.

- **The map generation system** — every dungeon in D2 is generated by a
  Binary Space Partitioning (BSP) algorithm seeded from a 32-bit game seed.
  This toolkit implements the algorithm and provides tools to read, predict,
  and analyze map layouts.

- **Function offsets for all major versions** — a cross-referenced table of
  virtual addresses for 80+ key functions across D2 v1.09d, v1.12a, v1.13c,
  and v1.14d, covering all major DLLs.

- **Data structures** — complete, annotated C struct definitions for
  `UnitAny`, `ItemData`, `MonsterData`, `PlayerData`, `StatList`, `Path`,
  `Room1`, `Room2`, and dozens more.

- **Ghidra automation scripts** — Python scripts that run inside Ghidra to
  automatically label functions, annotate struct accesses, and export symbols.

- **D2 Resurrected differences** — a dedicated reference covering the
  architectural changes in the 2021 remaster: 64-bit pointers, CASC archives,
  JSON data overrides, DirectX 12 hooking patterns, and more.

## Who Is This For?

This toolkit is for several distinct audiences. You do not need to be an
expert in all areas to use it productively.

### Modders
You want to change how the game behaves — add items, rebalance skills,
change monster stats, or add new mechanics. This toolkit explains exactly
which data tables control each behavior and provides Python scripts to
extract, modify, and analyze those tables from the game's MPQ archives.

### Emulator / Private Server Developers
You are building a server implementation compatible with the D2 client —
similar to PVPGN or D2GS. You need to understand the packet protocol
precisely enough to send and receive the correct bytes in the correct order.
The packet reference and sniffer in this toolkit give you that at the
command level.

### Bot / Automation Developers
You want to write a program that plays D2 automatically — navigating maps,
killing monsters, managing inventory, or farming specific items. This
requires reading the game's memory structures at runtime and understanding
which packets the client expects to receive and which it sends. This
toolkit documents both sides.

### Speedrunners and Researchers
You want to understand how the game's RNG works, predict item drops, find
optimal map seeds, or verify that two game sessions with the same seed
produce identical results. The seed tools and item roller are built for this.

### Reverse Engineering Students
Diablo II is an ideal case study in real-world RE: it uses a complex,
well-documented game protocol; its binary format is a classic example of
bit-packed data; its engine architecture demonstrates how a large C codebase
is organized across modular DLLs. This repository documents the process and
results of decades of community RE work.

### Tool Developers
You are building something adjacent to D2 — a character planner, loot
tracker, save editor, or item database — and you need machine-readable
definitions of the game's data formats.

## What Problems Does It Solve?

Before this toolkit, a researcher trying to understand D2's internals had to:

1. Hunt across dozens of scattered PhrozenKeep forum threads, some dating to
   2003, to find function offsets.
2. Download and cross-reference multiple community IDA databases of varying
   quality.
3. Manually decode `.d2s` files using an incomplete community wiki.
4. Reverse engineer the item generation pipeline from scratch, since no
   complete, code-level implementation existed in Python.
5. Write a packet sniffer from scratch with no documentation about what most
   packets mean.

D2RE consolidates all of that into one repository with consistent naming,
verified offsets, Python implementations, and plain-English documentation.

## Quick Start

If you just want to parse a save file and see what items your character has:

```bash
git clone https://github.com/yourname/d2re.git
cd d2re
pip install -r requirements.txt
python scripts/d2s_parser.py "C:/Users/You/Saved Games/Diablo II/YourChar.d2s"
```

If you want to watch live packets during a D2 game session:

```bash
# Linux/macOS (requires root for raw socket access)
sudo python scripts/packet_sniffer.py --interface eth0 --live --verbose

# Windows (run as Administrator, install Npcap first)
python scripts/packet_sniffer.py --interface "Ethernet" --live --verbose

# No network required — demo mode uses sample packets
python scripts/packet_sniffer.py --demo --verbose
```

If you want to extract all data tables from a D2 installation:

```bash
python scripts/mpq_extract.py --all-mpqs "C:/Diablo II/" --out ./tables/
```

### `packet_sniffer.py`

**What it does:** Captures raw TCP traffic on Diablo II's game ports (4000 for
direct connections, 6112 for Battle.net relay) and decodes every packet into a
human-readable, annotated stream. It does not just show you hex bytes — it
tells you what each packet *means* in game terms, what triggered it, what each
field value represents, and flags suspicious patterns.

**Why you need it:**

The D2 network protocol is undocumented by Blizzard. There are community
references (notably Jarulf's Guide and scattered forum posts) but no single,
comprehensive, code-level source. This tool lets you watch a game session in
real time and understand what is happening at the protocol level — essential
for server emulator development, bot development, or deep RE work.

**How it works:**

The tool uses [Scapy](https://scapy.net/) to capture raw TCP packets. Each
captured payload is passed through a lookup table of known packet definitions.
The first byte of every D2 game packet is the *command byte* — this uniquely
identifies the packet type. Once the command byte is known, the remaining bytes
are parsed according to the field definitions for that packet type.

A `GameStateTracker` object maintains a rolling snapshot of the observed game
session — the player's current HP/MP percentages, which skill is currently
active on each mouse button, the last seen unit modes, and recent ping timings.
This context is used to enrich the per-packet annotations. For example, when
a `RightSkillOnLocation` packet fires, the tracker already knows which skill
is on the right mouse button and includes it in the output.

**Usage:**

```bash
# Live capture on a network interface (requires admin/root)
sudo python scripts/packet_sniffer.py --interface eth0 --live

# Live capture with full per-packet explanations
sudo python scripts/packet_sniffer.py --interface eth0 --live --verbose

# Only show one specific command byte (e.g. watch only HP/MP updates)
sudo python scripts/packet_sniffer.py --interface eth0 --live --filter 0x0F

# Watch only skill casts
sudo python scripts/packet_sniffer.py --interface eth0 --live --filter 0x0C --verbose

# Save captured packets to JSON for offline analysis
sudo python scripts/packet_sniffer.py --interface eth0 --live --output session.json

# Read and decode a previously captured .pcap file
python scripts/packet_sniffer.py --pcap session.pcap --decode --verbose

# Filter a .pcap to only item action packets
python scripts/packet_sniffer.py --pcap session.pcap --filter 0x47 --hex

# Run demo mode — no network or admin required, uses hardcoded sample packets
python scripts/packet_sniffer.py --demo --verbose

# Generate C struct definitions for all known packets (pipe to a .h file)
python scripts/packet_sniffer.py --generate-structs > d2_packets.h

# List all known packets with their command bytes and categories
python scripts/packet_sniffer.py --list
```

**Sample output (verbose mode):**

```
14:23:05.441  C→S  0x0C  RightSkillOnLocation  9B  [skill]
  127.0.0.1:52341 → 192.168.1.1:4000
  unk                0x0000
  unk2               0x00
  x                  320        ← target X tile
  y                  288        ← target Y tile
  ╰─ Right mouse skill cast aimed at a tile coordinate. Fired on
     right-click for area skills: Teleport, Blizzard, Meteor, Frozen
     Orb, Bone Spirit, etc. Coordinates are tile-space, not pixel-space.
  → Active right-hand skill: Blizzard

14:23:05.449  S→C  0x0F  HpMpUpdate  5B  [combat]
  192.168.1.1:4000 → 127.0.0.1:52341
  hp_pct             24576      ← HP fraction: value/32768 = pct
  mp_pct             16384
  ╰─ Compact vitals update sent every time HP or MP changes. Displayed
     HP = (hp_pct / 32768) × MaxHP. High-frequency in combat.
  → HP: 75.0%  MP: 50.0%
```

**Output fields explained:**

- The timestamp (`14:23:05.441`) is wall-clock time of the captured packet.
- `C→S` / `S→C` indicates direction: Client to Server, or Server to Client.
- The hex code (`0x0C`) is the command byte — the first byte of the payload.
- The name (`RightSkillOnLocation`) is the human-readable packet name.
- The size in bytes.
- The category tag in brackets.
- Field table: field name, decoded value, and description.
- The italicized explanation block (verbose mode only).
- Context notes from the game state tracker.

**Understanding packet directions:**

In Diablo II, the client and server have a clear division of responsibility:

- **Client→Server (C2S):** The client tells the server what the *player
  intends* to do — move here, cast this skill, pick up that item, open this
  chest. The server is authoritative; it validates all C2S packets and may
  ignore or override them.

- **Server→Client (S2C):** The server tells the client what *happened* in the
  game world — this unit moved, that HP changed, this item appeared, that
  monster died. The client renders the game state it is told about; it does
  not decide what happened.

This means that in a legitimate game session, C2S packets will never directly
spawn items or teleport a player across the map — those decisions belong to
the server. Any C2S packet that attempts to bypass server validation is a
potential cheat attempt.

### `d2s_parser.py`

**What it does:** Parses a Diablo II `.d2s` character save file and extracts
every piece of data stored in it — character stats, skills, complete inventory
with all item properties, mercenary data, quest completion flags, and waypoints.
It handles the game's unusual bit-packed encoding and the variable-width
item data format.

**Why you need it:**

The `.d2s` format is not documented by Blizzard. It uses a custom bit-stream
encoding where different fields occupy different numbers of bits, packed
densely together with no alignment padding. A naive hex editor will show you
nothing useful. This parser implements the full decoding logic, including the
variable-width per-stat fields, the item bit-stream, and the checksum algorithm.

**How D2 save files are structured:**

A `.d2s` file has several distinct sections, each identified by a 2–4 byte
magic number:

```
[Fixed header]  0x00–0x14B  — name, class, level, timestamps, hotkeys, seed
[Quest data]    magic "Woo!" — which quests are complete per difficulty
[Waypoints]     magic "WS"  — which waypoints are activated per difficulty
[NPC flags]     magic 0x0177 — which NPC dialogues have been heard
[Stats]         magic "gf"  — character stats in bit-stream format
[Skills]        magic "if"  — 30 skill allocation bytes
[Items]         magic "JM"  — full item list in bit-stream format
[Corpse items]  magic "JM"  — items left on a death corpse, if any
[Merc items]    magic "jf"  — mercenary's equipped items
[Iron Golem]    (Necro only) — the item consumed to create an Iron Golem
```

The stats section uses variable-width bit fields: each stat entry starts
with a 9-bit stat ID, followed by N bits of value where N is defined in the
game's `itemstatcost.txt` data table (different for each stat). The section
ends when the 9-bit ID equals `0x1FF` (all bits set).

The items section is even more complex: each item record varies in length
depending on the item type, quality, number of affixes, and whether the
item is socketed (in which case each socketed gem/rune is a full sub-record).

**Usage:**

```bash
# Basic: print character summary and item list
python scripts/d2s_parser.py "C:/Users/You/Saved Games/Diablo II/HardcoreBarb.d2s"

# Output as JSON (useful for piping to other tools)
python scripts/d2s_parser.py MyChar.d2s --json

# Show all items in detail
python scripts/d2s_parser.py MyChar.d2s --items

# Verify the save file checksum (detects corruption or external edits)
python scripts/d2s_parser.py MyChar.d2s --verify
```

**What "ilvl" means:**

Item level (iLvl) is the internal property that determines which affixes can
appear on an item. It is not the same as the character level required to use
the item. A higher iLvl means higher-tier affixes are possible. For monster
drops, iLvl ≈ monster level. For gambling, iLvl = character level ± 5.

**What the item quality names mean:**

| Quality | How it's generated | Examples |
|---|---|---|
| Inferior | Penalty roll — slightly worse stats | Cracked Short Sword |
| Normal | No affixes | Short Sword |
| Superior | Bonus roll — slightly better base stats | Superior Short Sword (+% ED) |
| Magic | 0–1 prefix + 0–1 suffix from magic tables | Wounding Short Sword of Quickness |
| Rare | Up to 6 affixes from a larger pool, with a procedural name | Death Blade |
| Set | Predefined stats + set bonuses when multiple pieces are worn | Sigon's Blade |
| Unique | Predefined, fixed stats; one per base item type | Doombringer |
| Crafted | Fixed recipe bonuses + random magic affixes | Blood Sword |

### `mpq_extract.py`

**What it does:** Extracts data from Diablo II's MPQ archive files (Classic)
or CASC storage (D2 Resurrected). MPQ stands for "Mike O'Brien Pack" — it is
Blizzard's proprietary archive format. All of D2's data tables, sprites,
sounds, strings, and maps are stored in MPQ files.

**Why you need it:**

To analyze or modify any aspect of D2's game data, you need to access the
source files. The most important of these are the `.txt` data tables in
`data/global/excel/` — tab-separated files that define every item, monster,
skill, treasure class, and level in the game. This script extracts them to
CSV or JSON for easy analysis.

**What's in the MPQ archives:**

D2 ships with several MPQ files that are loaded in order (later files override
earlier ones):

- `d2data.mpq` — core game data: all `.txt` tables, `.tbl` string tables,
  `.ds1`/`.dt1` map files, base sprites
- `d2exp.mpq` — Lord of Destruction expansion data: adds Act 5, new classes,
  new items, new monsters
- `patch_d2.mpq` — patches applied over the base game; always loaded last
  and takes highest priority

**The data tables explained:**

Each `.txt` file in `data/global/excel/` is a tab-separated table where the
first row is column headers and the second row is column type descriptors
(which the game uses internally but you can ignore). The most important tables:

| File | What it defines |
|---|---|
| `weapons.txt` | Every weapon: base damage, speed, strength/dex requirements, socket limits |
| `armor.txt` | Every armor/helm/glove/boot/belt/shield: base defense, requirements |
| `misc.txt` | All other items: rings, amulets, charms, potions, scrolls, runes, gems |
| `skills.txt` | Every skill: mana cost, cooldown, damage formulas, synergy links |
| `monstats.txt` | Every monster: HP, damage, speed, resistances, experience, AI code |
| `levels.txt` | Every level/area: dimensions, monster types, waypoint flags, area level |
| `treasureclassex.txt` | Loot tables: which items drop from which monsters in which quantities |
| `itemstatcost.txt` | Every item property: bit width in save files, display formulas |
| `magicprefix.txt` | All magic/rare prefixes: which stats they add and at what rolls |
| `magicsuffix.txt` | All magic/rare suffixes |
| `uniqueitems.txt` | Every unique item: its exact fixed properties |
| `setitems.txt` | Every set item: partial and full set bonuses |
| `cubemain.txt` | Every Horadric Cube recipe: ingredients, outputs, conditions |
| `runes.txt` | Runeword definitions: rune sequences and resulting properties |
| `hireling.txt` | Mercenary types: starting stats, skill progression, equipment |
| `experience.txt` | Experience thresholds for each level for each class |

**Usage:**

```bash
# Extract all data tables from a D2 installation to CSV files
python scripts/mpq_extract.py --all-mpqs "C:/Diablo II/" --out ./tables/

# Extract a single table and print the first 10 rows
python scripts/mpq_extract.py --mpq d2data.mpq --table weapons

# Extract a table to CSV for use in Excel or Pandas
python scripts/mpq_extract.py --mpq d2data.mpq --table monstats --csv

# Extract a table to JSON for use in JavaScript or Python
python scripts/mpq_extract.py --mpq d2data.mpq --table skills --json-out

# Build the Treasure Class tree for drop simulation
python scripts/mpq_extract.py --all-mpqs "C:/Diablo II/" --tc-tree

# Extract all string table entries to JSON
python scripts/mpq_extract.py --all-mpqs "C:/Diablo II/" --strings

# Extract D2R data via CASC (requires pycasc)
python scripts/mpq_extract.py --d2r "C:/Program Files (x86)/Diablo II Resurrected/" --out ./d2r_tables/
```

**Scenario: I want to find every item that can roll +2 to all skills**

```bash

python scripts/mpq_extract.py --all-mpqs "C:/Diablo II/" --table magicprefix --csv
python scripts/mpq_extract.py --all-mpqs "C:/Diablo II/" --table magicsuffix --csv

# Then in Python:
import csv

    for row in csv.DictReader(f):

            print(row['Name'], row.get('mod1code'), row.get('mod1param'))
```

### `item_roller.py`

**What it does:** Simulates the Diablo II server-side item generation pipeline
in Python. Given a seed and item level, it reproduces the exact same quality
rolls and affix selections that the D2 game server would make. It also supports
Monte Carlo simulation — running millions of drop attempts to compute actual
drop rates for specific items or quality tiers.

**Why you need it:**

D2's item generation is deterministic. The same seed always produces the same
item. This is how bots can "seed check" — test a game's seed to see if a
specific unique item will drop at a specific monster. It's also why experienced
players can run specific areas on specific characters and get predictable results.

**The six stages of item generation (in order):**

**Stage 1 — Treasure Class Resolution:**
When a monster dies, it does not have a fixed item in mind. Instead, it has a
"Treasure Class" (TC) — a named pool of possible drops. The TC system is
recursive: a TC entry can point to another TC, not just a final item. The game
keeps resolving until it reaches a base item code or a "NoDrop" result.

For example, `tc/Act 5 Super C` → `tc/Weap87` → with probability X picks
`Phase Blade (7cr)`. The `NoDrop` probability determines how often nothing
drops at all. Magic Find % increases the probability of reaching high-quality
TC entries rather than increasing the overall drop rate.

**Stage 2 — Item Level (iLvl) Assignment:**
Once the base item type is known, the item level is calculated:
- Monster drops: `iLvl = min(monster_level + boss_bonus, area_level × 2)`
- Area objects (chests): `iLvl = area_level`
- Gambling: `iLvl = character_level + 5` (but affixes capped between clvl-5 and clvl+4)
- Crafting: `iLvl = floor(clvl/2) + floor(ingredient_ilvl/2)`

iLvl matters because every affix has a minimum item level requirement. Higher
iLvl = access to higher-tier, more powerful affixes.

**Stage 3 — Quality Determination:**
The server rolls against a series of thresholds in order: Unique → Set → Rare
→ Magic → Superior → Normal → Inferior. The thresholds are calculated as:

```
threshold = base_ratio × (effective_MF + base_offset) / 1024
```

Magic Find (MF%) has diminishing returns for high-quality items. At 300% raw
MF, the effective MF for Unique items is only 150%, for Set items 273%, for
Rare items 310%. Magic items have no diminishing returns.

**Stage 4 — Affix Selection:**
For magic items: up to 1 prefix and 1 suffix are selected independently.
Each is filtered by `iLvl ≥ affix.level` and item type compatibility (you
cannot get +% Enhanced Damage on a helmet, for example). The selection
is random but weighted by each affix's "frequency" column in the data tables.

For rare items: a pool of up to 3 prefixes and 3 suffixes is built, then 3–6
affixes are drawn from it without replacement.

**Stage 5 — Unique/Set Matching:**
If quality = Unique, the game looks for a unique item with the same base code
where `iLvl ≥ unique.level`. If none qualify, the item downgrades to Rare.
This is why high-level characters can find uniques that low-level characters
cannot — the Grandfather (Colossus Blade) requires iLvl 65.

**Stage 6 — Sockets, Ethereal, and Personalization:**
Socket count is rolled from dice bounded by `items.txt` MaxSock columns.
Ethereal items (25% bonus to defense/damage, cannot be repaired) have a
1-in-17 chance on eligible item types, but only for monster drops — shops
and gambling never produce ethereal items.

**Usage:**

```bash

# Roll a specific seed and see what quality it produces

# Simulate drops from a Treasure Class (requires tc_tree.json from mpq_extract)
python scripts/item_roller.py --tc "Act 5 Super C" --runs 500000 --mf 400
```

### `map_seed_tool.py`

**What it does:** Reads map seeds from `.d2s` save files, predicts dungeon
room layouts using a simulation of D2's BSP algorithm, visualizes maps as
ASCII art, and brute-forces seeds matching custom criteria.

**Why you need it:**

Every Diablo II game has a single 32-bit map seed stored in the save file.
This seed deterministically generates every dungeon, outdoor area, and map
layout in the game. Two characters with the same map seed will have identical
maps. This is the foundation of D2's "map hacking" — pre-generating the map
before entering an area to know where everything is.

**How the seed cascade works:**

```
game_seed (32-bit)
  → act_seed[0] = LCG(game_seed, 1)       ← Act 1 seed
  → act_seed[1] = LCG(game_seed, 2)       ← Act 2 seed
  ...
  → level_seed[n] = LCG(act_seed, levelId)  ← per-area seed
      → room_seed[n] = LCG(level_seed, roomIndex)  ← per-room seed
```

LCG stands for Linear Congruential Generator — a simple PRNG with the formula:
`next = (prev × 0x6AC690C5 + 1) mod 2^32`

All maps, monster spawns, chest contents, and waypoint positions for a given
game session are deterministic from the single 32-bit game seed.

**Usage:**

```bash
# Read map seed from a save file and show per-act derived seeds
python scripts/map_seed_tool.py --d2s MyChar.d2s --show-seed

# Predict room layout for a specific level and render ASCII map
python scripts/map_seed_tool.py --seed 0x3F7A1B2C --level 21 --ascii

# List all predicted rooms with coordinates
python scripts/map_seed_tool.py --seed 0x3F7A1B2C --level 21 --rooms

# Brute-force seeds for levels with unusual room layouts
python scripts/map_seed_tool.py --level 21 --brute --max 1000000

# Combine: read seed from save and render map for Pit Level 1
python scripts/map_seed_tool.py --d2s MyChar.d2s --level 15 --bsp --ascii
```

## Understanding D2's Engine Architecture

Diablo II (classic, v1.09d–v1.13c) splits its engine across 12 DLLs, each
with a distinct responsibility. Understanding which module owns which feature
is the first step in any RE task.

```
User Input
    ↓
D2Win.dll ─── reads keyboard/mouse ──────────────────────────┐
    ↓                                                          │
D2Client.dll ─── interprets input ─── sends C2S packets ──→ D2Net.dll
    ↑                                                          ↓
    │          receives S2C packets ←──────────────────── D2Game.dll
    │                                                          │
    ├── D2Gfx.dll ←── D2CMP.dll (sprites) ──────────────────  │
    ├── D2Sound.dll ←── audio events ───────────────────────── │
    └── D2Lang.dll ←── string lookups ─────────────────────── │
                                                               │
D2Common.dll ←── shared data / stat engine ──────────────────┘
     ↑
     └── Fog.dll (memory) + Storm.dll (MPQ) + D2Lang.dll (strings)
```

In single-player, D2Game.dll runs in the same process as D2Client.dll — the
server and client are co-located, communicating through a loopback mechanism
rather than a real network socket.

In multiplayer (open Battle.net or TCP/IP), D2Net.dll mediates between
D2Client.dll and the actual network layer, handling packet buffering,
sequencing, and the simple acknowledgment protocol.

## Understanding the Network Protocol

D2 uses a custom binary protocol over TCP. There is no header with a length
field — each packet type has either a fixed size (known from its command byte)
or a variable size (which must be parsed incrementally). The protocol is
session-stateful: the meaning of some packet fields depends on previously
established game context.

**The 25 Hz server tick:** D2's game server runs at 25 frames per second.
All timers, cooldowns, movement speeds, and damage-over-time durations are
measured in server ticks (1 tick = 40ms). When you see a latency value in-game,
it represents the number of milliseconds for a round-trip ping — if your ping
is 100ms, you are roughly 2.5 ticks behind the server.

**Sequence numbers:** D2Net wraps game packets in a lightweight envelope with
a 2-byte sequence number. Out-of-order packets are buffered and reordered.
Duplicate sequence numbers are silently dropped. This provides a reliable
ordered channel over TCP, which is already reliable, so in practice sequence
numbers matter most during connection establishment.

**Packet inspection for RE purposes:** The best way to understand a specific
game mechanic at the protocol level is to reproduce the action in-game while
running the sniffer, then examine the packet sequence. For example, to
understand how the Horadric Cube transmutation works:

1. Start the sniffer: `sudo python scripts/packet_sniffer.py --interface eth0 --live --verbose`
2. Put ingredients in the cube and click Transmute.
3. Examine the captured packets — you will see `C2S CubeTransmute (0x21)` followed
   by one or more `S2C ItemActionOwned (0x4F)` packets for the output item.

## Understanding Save Files

The `.d2s` format was designed in 2000 for compact storage on 56K-modem-era
computers. Every bit is deliberately packed to minimize file size. The format
has never been officially documented — everything here was reverse engineered
from the binary.

**Bit ordering:** D2 uses LSB-first (least significant bit first) bit packing
throughout. If you are reading the save format manually with a hex editor,
you must reverse the bit order within each byte before interpreting multi-bit
fields. The `BitReader` class in `d2s_parser.py` handles this automatically.

**Fixed-point math:** Hit points, mana, and stamina are stored as 21-bit
fixed-point values with 8 bits of fractional precision. To get the displayed
value, divide by 256. This allows the game to track fractional HP internally
(from regeneration, life-per-second items, etc.) while displaying whole numbers
to the player.

**The item bit-stream:** Item records have no fixed size. The bit-stream parser
must read each field in order and use the values it reads to determine how many
bits come next. For example, the `Identified` flag (1 bit) tells the parser
whether the affix data section will follow. The `Socketed` flag tells it whether
to expect sub-item records after the main record. This makes the format
extremely compact but also difficult to parse without a complete state machine.

## Understanding Item Generation

The complete 6-stage pipeline is implemented in `scripts/item_roller.py` and
documented in `references/item-generation.md`. The most important concept for
practical use is the interaction between Magic Find and quality thresholds.

**Magic Find diminishing returns explained:**

Raw MF% is the stat shown on your character screen. Effective MF% is what
the game actually uses when rolling quality. The relationship is:

```
Effective MF (Unique) = RawMF × 250 / (RawMF + 250)

Examples:
  100% raw MF → 71% effective for Unique rolls
  200% raw MF → 111% effective
  300% raw MF → 150% effective
  500% raw MF → 167% effective
  1000% raw MF → 200% effective
```

The curve flattens quickly — going from 300% to 400% raw MF adds only 10%
effective MF for Unique rolls. This is why experienced players target
200–350% MF and optimize other stats rather than stacking MF beyond that.

## Understanding Map Generation

D2's DRLG (Dungeon Random Level Generator) has three modes: preset, outdoor,
and maze. The maze algorithm is the most complex and uses recursive Binary
Space Partitioning.

**BSP explained for non-CS readers:**

Imagine you have a rectangular region of graph paper representing a dungeon
floor. You want to fill it with connected rooms. BSP works like this:

1. Pick a random point along one edge and draw a line through the rectangle,
   splitting it into two smaller rectangles.
2. Repeat for each smaller rectangle (recursively), until the rectangles are
   too small to split further.
3. Each final (unsplit) rectangle becomes a room. The room is slightly smaller
   than the rectangle that contains it, with random placement within it.
4. Connect each pair of sibling rooms (rooms that share a parent in the BSP
   tree) with an L-shaped hallway.

The `map_seed_tool.py` implements this algorithm using the same LCG as the
game engine, so given the same seed it produces the same room layout that
the actual game would generate.

## Frequently Asked Questions

**Q: Does this work with the original CD version of D2?**

Yes, for versions 1.09d through 1.14d. Function offsets are provided for all
four major versions. The 1.14d offsets are for the single merged executable.

**Q: Does this work with Diablo II: Resurrected (D2R)?**

The Python tools (d2s_parser, mpq_extract) work with D2R save files and CASC
archives respectively. The function offsets and C struct sizes are different
in D2R (64-bit). See `references/d2r-resurrected.md` for D2R-specific guidance.

**Q: Can I use this to cheat in online games?**

This toolkit is designed for reverse engineering, research, and private
server development. Using it to gain unfair advantages in official online games
(Battle.net) violates Blizzard's Terms of Service and may result in a permanent
ban. We do not support that use case.

**Q: What's the difference between packet sniffing and memory reading?**

Packet sniffing reads data from the network — the bytes being sent and
received between the client and server. Memory reading reads data directly
from the game process's RAM. Both are useful for different purposes: packet
sniffing is non-invasive and shows you exactly what the client and server
are communicating; memory reading gives you access to the full game state
including things the network protocol abstracts away.

**Q: Why are some packet fields labeled "unk"?**

"unk" stands for "unknown" — a field whose purpose has not been conclusively
determined through reverse engineering. Community RE work is ongoing and some
fields remain understood only by their size, not their purpose. If you figure
out what an "unk" field does, please open a PR.

**Q: How do I find my network interface name?**

```python
# Run in Python to list available interfaces
from scapy.all import get_if_list
print(get_if_list())
```

**Q: The sniffer shows "unknown packet" for some bytes — why?**

Not all packet types are fully documented. D2 has approximately 180 packet
types; this toolkit documents the most common and most important ones.
"Unknown" packets are those without a matching entry in the lookup tables.
The raw hex is still printed so you can analyze them manually.

**Q: Can I use these scripts with other ARPGs?**

No — the packet definitions, save file format, item generation logic, and
struct layouts are all specific to Diablo II. The Python patterns (bit reader,
pattern scanner, etc.) could be adapted for other games.
